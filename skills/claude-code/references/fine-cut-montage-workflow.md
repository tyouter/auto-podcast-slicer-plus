# 一期精剪 / 混剪执行（工作流1）—— 去废料保主体 + preview 高光片头

整期长素材 → 一条连续成片：剔除废料、保留主体、可加高光预告片头。执行层是 garden_core 的
`run_montage`（N 区间按指定顺序拼一条连续长片，字幕时间轴自动偏移合并）。

## 标准步骤

1. **全片转录**（`run_from_audio` 转录段 / `FunASRLocal`）。
   - ⚠️ 长音频不会 OOM：`FunASRLocal` 已内建 **VAD 静音对齐切块 + 块间 `gc.collect()+torch.cuda.empty_cache()`**（块边界落静音里，句子不被切断）。86min 实测 17 块、2019 段、标点 100%、末段对齐到片尾、~250s 转完。**不要手动盯 OOM、不要降 numpy** —— 代码已保证。
   - 存完整 `transcript.pkl`（供 `run_montage` 复用，不重复转录）+ 可读 `transcript.json`（供制作人读）。

2. **制作人读全片标废料**（创作判断，不是机械）。高效读法：用 execute_code 把连续同说话人的短 segment **合并成段落**再输出（`[mm:ss] spk> 合并文本`），分前后两半读，token 省、脉络清。
   - 废料典型形态（≠ 长静音，gap 往往很少）：**开头准备**（设备调试/"已经在录了"/反复商量开头/演练自我介绍 —— 可能长达近 10 分钟，到一句明确的「ok 卡，大家好…」才正式开场）、**中场设备调整**（暂停挪伞/聊灯光/"聊了多久了"）、结尾**半句话戛然而止**（要往前找完整句收尾）。
   - 大静音 `gap>4s` 扫描可辅助定位段落分界，但"未准备好/跑题"是语义信号，必须通读。

3. **出创作蓝图 → 用户确认**（skill 工作流0 阶段2，强制）。给**有判断的推荐**而非选项菜单：连贯思想对话 → 线性精剪（不打碎重组）；明确把 1–2 个真正需要用户拍板的创作点单列（开头某段留作彩蛋否、结尾收法）。

4. **preview 高光片头**（可选，强力开场）：从全片挑 7–8 个**最浓缩、≤5s、一句话**的金句，每个一个 CutPoint（精确到单 segment 时间戳），快剪拼在正片前。约 7 句 × 4s ≈ 30s。先给金句**清单**让用户增删，确认后再去 transcript 精确定位时间戳。

5. **`run_montage` 渲染**：组装 cut_points 列表 = `[preview 7 句 → 正片段1 → 正片段2 → 收尾段]`。
   - **输出顺序 = cut_points 列表顺序**，可与原片时间序不同 —— 收尾金句（如"漂流瓶"段）在原片靠前，只需把它的 CutPoint **列在最后**即可移到片尾。
   - 4K 选 `RenderOptions(horizontal_width=3840, horizontal_height=2160)`；混剪长片通常只要横版，`render_vertical=False` 省掉无用竖版渲染。

6. **render_gate → quality-audit（每条全审）→ NAS 交付**（本地 SSD 渲完再 robocopy，NAS 端逐字节核对）。

## 验证（自己跑，不信自报）

`run_montage` 的功能验证用最小用例独立跑：从 transcript 取 3 个区间（各 ~10s，故意把"原片靠后但要输出靠前"的一段制造乱序）→ 检查：输出一条 mp4、时长≈区间和、字幕单调无重叠、**末段字幕文字 = 排在最后那个 CutPoint 对应原片区间的内容**（证明乱序拼接正确）。小分辨率（960×540）+ 短区间，~1 分钟即可验证拼接逻辑，无需跑全片 4K。

## 勘误 / 隐名

姓名隐去、转录错字纠正走 `PipelineOptions.errata`（ErrataConfig）。勘误表是用户定的，**只按用户给的条目建，不自作主张增减**。典型：把一个人的真名（及其所有转录错写变体）统一替换成代称；个别专有名词错字（如鸟名）纠正。哪些人隐名、哪些保留真名由用户逐一确认。
